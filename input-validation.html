Input Validation & Sanitization Chapter 4 ​

Series Navigation

← Chapter 3: Authentication & Passwords · Chapter 5: Security Headers & CSP →

The single most repeated rule in web security is: never trust user input. This means anything your application receives — form fields, URL parameters, headers, file names, JSON payloads — must be treated as potentially hostile until validated and sanitized. In this chapter, we tackle the three most common and dangerous input-based attacks.

The Three Threats ​

AttackWhat happensImpact
XSSAttacker injects scripts that run in other users' browsersSession theft, account takeover, data exfiltration
SQL InjectionAttacker manipulates database queries via crafted inputData breach, authentication bypass, full DB takeover
CSRFAttacker tricks a logged-in user's browser into making unauthorized requestsUnwanted actions performed on behalf of the victim

Part 1 — Cross-Site Scripting (XSS) ​

How XSS Works ​

html
<!-- Imagine this user input is rendered directly into the page: -->
<p>Hello, <script>document.cookie // sent to attacker</script>!</p>

If your application inserts unsanitized user-controlled content into the DOM, the injected script runs with full access to the page — cookies, tokens, and all.

Step 1: Use a Templating Engine That Auto-Escapes ​

Modern template engines like Handlebars, EJS, and Nunjucks auto-escape HTML by default. Always use their safe output syntax:

handlebars
<!-- ✅ Auto-escaped: <script> becomes &lt;script&gt; -->
<p>Hello, {{ username }}</p>

<!-- ❌ Raw/unescaped output — only use for trusted HTML you control -->
<p>{{{ trustedHtmlContent }}}</p>

Step 2: Sanitize Rich HTML Input with DOMPurify ​

When you genuinely need to accept rich HTML (e.g., a blog editor), sanitize it before storing or rendering it:

bash
npm install isomorphic-dompurify
javascript
import DOMPurify from 'isomorphic-dompurify'

export function sanitizeHtml(dirty) {
  return DOMPurify.sanitize(dirty, {
    ALLOWED_TAGS: ['b', 'i', 'em', 'strong', 'a', 'p', 'ul', 'ol', 'li', 'br'],
    ALLOWED_ATTR: ['href', 'title', 'target'],
    ALLOW_DATA_ATTR: false
  })
}

Step 3: Validate and Sanitize All Inputs with Zod ​

Schema validation with Zod lets you enforce the exact shape, type, and constraints of every input — server-side, before it ever touches your business logic:

bash
npm install zod
javascript
import { z } from 'zod'

const RegisterSchema = z.object({
  email: z.string().email().max(254).toLowerCase().trim(),
  password: z.string().min(12).max(72),
  username: z
    .string()
    .min(3)
    .max(32)
    .regex(/^[a-zA-Z0-9_-]+$/, 'Username may only contain letters, numbers, _ and -')
})

// In your route:
export function validateBody(schema) {
  return (req, res, next) => {
    const result = schema.safeParse(req.body)
    if (!result.success) {
      return res.status(400).json({ errors: result.error.flatten() })
    }
    req.body = result.data // Replace body with parsed, safe data
    next()
  }
}

router.post('/register', validateBody(RegisterSchema), async (req, res) => {
  // req.body is now guaranteed to match RegisterSchema exactly
})

Validate everywhere

Validate inputs on:

  • Route handlers — using a middleware like the one above
  • Database layer — use your ORM's schema constraints as a last line of defense
  • Client-side — for UX only, never as your only defense. Server-side validation is mandatory.

Part 2 — SQL Injection ​

How SQL Injection Works ​

javascript
// ❌ NEVER do this — string concatenation with user input
const query = `SELECT * FROM users WHERE email = '${req.body.email}'`
// Attacker sends: ' OR '1'='1
// Query becomes: SELECT * FROM users WHERE email = '' OR '1'='1'
// Returns ALL users!

Step 1: Always Use Parameterized Queries ​

The fix is to never concatenate user input into SQL. Use parameterized queries (also called prepared statements), which keep code and data completely separate:

javascript
// ✅ Using the 'pg' library (PostgreSQL)
const { rows } = await pool.query(
  'SELECT * FROM users WHERE email = $1',
  [req.body.email] // Passed as a parameter, never interpolated
)

// ✅ Using the 'mysql2' library
const [rows] = await db.execute(
  'SELECT * FROM users WHERE email = ?',
  [req.body.email]
)

The database driver handles escaping. The user input can never alter the query structure.

Step 2: Use an ORM with Built-in Protection ​

ORMs like Prisma or Drizzle use parameterized queries under the hood for all standard operations:

javascript
// ✅ Prisma — always safe for standard queries
const user = await prisma.user.findUnique({
  where: { email: req.body.email }
})

// ✅ Raw Prisma queries still need parameterization
const result = await prisma.$queryRaw`
  SELECT * FROM users WHERE email = ${req.body.email}
`
// Prisma's tagged template literal handles parameterization automatically

Beware of raw query escape hatches

ORMs often provide a way to run raw SQL. Prisma has $queryRawUnsafe, Drizzle has sql.raw. Never pass user input into these — they bypass all protections.

Step 3: Principle of Least Privilege for DB Users ​

Your application's database user should only have the permissions it actually needs:

sql
-- Create a restricted app user
CREATE USER app_user WITH PASSWORD 'strong_random_password';

-- Grant only what's needed
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO app_user;

-- Never grant DROP, CREATE, ALTER, or superuser privileges to the app

Part 3 — Cross-Site Request Forgery (CSRF) ​

How CSRF Works ​

When a user is logged in to bank.com, their browser automatically sends authentication cookies with every request to bank.com — including requests triggered by a malicious third-party page:

html
<!-- On evil.com — the user doesn't even see this: -->
<img src="https://bank.com/transfer?to=attacker&amount=5000" />

The bank sees a valid, authenticated request and processes the transfer.

Step 1: Use SameSite Cookie Attribute ​

The most effective modern defense. Set your session/JWT cookies with SameSite=Strict or SameSite=Lax:

javascript
res.cookie('token', jwt, {
  httpOnly: true,    // ✅ Not accessible to JavaScript
  secure: true,      // ✅ Only sent over HTTPS
  sameSite: 'strict' // ✅ Never sent on cross-site requests
})
SameSite ValueBehavior
StrictCookie never sent on any cross-site request (most secure)
LaxCookie sent on top-level navigation (GET), not on sub-resource requests
NoneAlways sent — requires Secure, defeats CSRF protection

Step 2: CSRF Tokens for State-Changing Requests ​

For extra defense-in-depth (or when you can't use SameSite), use CSRF tokens:

bash
npm install csrf-csrf
javascript
import { doubleCsrf } from 'csrf-csrf'

const { generateToken, doubleCsrfProtection } = doubleCsrf({
  getSecret: () => process.env.CSRF_SECRET,
  cookieName: '__Host-psifi.x-csrf-token',
  cookieOptions: { secure: true, sameSite: 'strict' }
})

// Expose the token to your frontend
app.get('/csrf-token', (req, res) => {
  res.json({ csrfToken: generateToken(req, res) })
})

// Protect state-changing routes
app.use('/api', doubleCsrfProtection)

In your frontend, send the token with every mutating request:

javascript
const { csrfToken } = await fetch('/csrf-token').then(r => r.json())

await fetch('/api/transfer', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'x-csrf-token': csrfToken
  },
  body: JSON.stringify({ to: 'friend', amount: 50 })
})

Do I need CSRF tokens if I use SameSite cookies?

If all your auth cookies are SameSite=Strict or SameSite=Lax, CSRF tokens are largely redundant for modern browsers. However, older browsers and some edge cases (custom schemes, native apps) don't respect SameSite. CSRF tokens provide a meaningful extra layer for critical actions like fund transfers or account deletion.

✅ Chapter 4 Checklist ​

  • [ ] All HTML output is auto-escaped by your template engine
  • [ ] Rich HTML input sanitized with DOMPurify before storage
  • [ ] All request inputs validated against a strict schema (Zod or equivalent)
  • [ ] Zero string concatenation in SQL queries — parameterized everywhere
  • [ ] ORM raw query escape hatches never receive user input
  • [ ] Database user has minimum required privileges
  • [ ] Session/JWT cookies set with httpOnly, secure, sameSite: 'strict'
  • [ ] CSRF tokens implemented for critical state-changing actions
