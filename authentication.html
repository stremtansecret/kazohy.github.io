Authentication & Passwords Chapter 3 ​

Series Navigation

← Chapter 2: HTTPS & Transport Security · Chapter 4: Input Validation & Sanitization →

Authentication is the front door of your application. It's also one of the most frequently broken things in web security. In this chapter we'll cover the right way to store passwords, how to issue and verify JWTs safely, and how to layer multi-factor authentication (MFA) on top for maximum protection.

The Golden Rule: Never Store Plain-Text Passwords ​

If your database is ever breached, plain-text passwords mean every user's account on every site they reuse that password is compromised instantly. This is not hypothetical — it happens constantly.

The solution is one-way hashing with a slow, salted algorithm.

Step 1: Hash Passwords with bcrypt ​

bcrypt is the industry standard for password hashing. It's deliberately slow (configurable via a "cost factor"), making brute-force attacks computationally expensive.

bash
npm install bcrypt

Hashing a password on registration:

javascript
import bcrypt from 'bcrypt'

const SALT_ROUNDS = 12 // Cost factor: higher = slower = safer. 12 is a good balance.

export async function hashPassword(plainText) {
  return bcrypt.hash(plainText, SALT_ROUNDS)
}

Verifying a password on login:

javascript
export async function verifyPassword(plainText, storedHash) {
  return bcrypt.compare(plainText, storedHash)
  // Returns true/false — bcrypt extracts the salt from the stored hash automatically
}

Why not SHA-256 or MD5?

General-purpose hash functions (MD5, SHA-1, SHA-256) are designed to be fast — great for checksums, terrible for passwords. A GPU can compute billions of SHA-256 hashes per second. bcrypt at cost factor 12 takes ~300ms per hash, making brute-force attacks ~10 million times harder.

Other solid choices: Argon2id (winner of the Password Hashing Competition) and scrypt.

Understanding salts

A salt is a random value generated per password and mixed into the hash. This means:

  • Two users with the same password get different hashes
  • Pre-computed rainbow table attacks are useless
  • Attackers must brute-force each hash individually

bcrypt generates and stores the salt automatically — you don't manage it manually.

Step 2: Build a Secure Registration & Login Route ​

javascript
// src/routes/auth.js
import { Router } from 'express'
import { hashPassword, verifyPassword } from '../lib/password.js'
import { createUser, findUserByEmail } from '../db/users.js'
import { signToken } from '../lib/jwt.js'

const router = Router()

// POST /auth/register
router.post('/register', async (req, res, next) => {
  try {
    const { email, password } = req.body

    // ✅ Validate input length — bcrypt silently truncates at 72 bytes
    if (!password || password.length < 12 || password.length > 72) {
      return res.status(400).json({ error: 'Password must be 12–72 characters.' })
    }

    const existingUser = await findUserByEmail(email)
    if (existingUser) {
      // ✅ Don't reveal whether an email exists — use a generic message
      return res.status(400).json({ error: 'Registration failed. Please try again.' })
    }

    const hash = await hashPassword(password)
    const user = await createUser({ email, hash })

    res.status(201).json({ message: 'Account created.' })
  } catch (err) {
    next(err)
  }
})

// POST /auth/login
router.post('/login', async (req, res, next) => {
  try {
    const { email, password } = req.body

    const user = await findUserByEmail(email)

    // ✅ Always run bcrypt.compare even if the user doesn't exist
    // This prevents timing attacks that reveal valid email addresses
    const dummyHash = '$2b$12$invalidhashfortimingprotection000000000000000000000'
    const hash = user?.hash ?? dummyHash

    const valid = await verifyPassword(password, hash)

    if (!user || !valid) {
      return res.status(401).json({ error: 'Invalid email or password.' })
    }

    const token = signToken({ sub: user.id, email: user.email })
    res.json({ token })
  } catch (err) {
    next(err)
  }
})

export default router

Step 3: Issue and Verify JWTs Safely ​

JSON Web Tokens (JWTs) are a stateless way to authenticate API requests. A signed JWT lets your server verify the user's identity without querying a session database on every request.

bash
npm install jsonwebtoken
javascript
// src/lib/jwt.js
import jwt from 'jsonwebtoken'

const SECRET = process.env.JWT_SECRET       // Must be a long, random string
const ISSUER = process.env.JWT_ISSUER ?? 'secure-site'

export function signToken(payload) {
  return jwt.sign(payload, SECRET, {
    expiresIn: '15m',  // ✅ Short-lived access tokens
    issuer: ISSUER,
    algorithm: 'HS256'
  })
}

export function verifyToken(token) {
  return jwt.verify(token, SECRET, {
    issuer: ISSUER,
    algorithms: ['HS256'] // ✅ Explicitly whitelist — prevents the "alg: none" attack
  })
}

Generate a secure secret:

bash
node -e "console.log(require('crypto').randomBytes(64).toString('hex'))"

Add it to .env:

ini
JWT_SECRET=your_generated_64_byte_hex_string_here
JWT_ISSUER=secure-site

An auth middleware to protect routes:

javascript
// src/middleware/requireAuth.js
import { verifyToken } from '../lib/jwt.js'

export function requireAuth(req, res, next) {
  const authHeader = req.headers['authorization']
  const token = authHeader?.startsWith('Bearer ') ? authHeader.slice(7) : null

  if (!token) {
    return res.status(401).json({ error: 'Authentication required.' })
  }

  try {
    req.user = verifyToken(token)
    next()
  } catch {
    return res.status(401).json({ error: 'Invalid or expired token.' })
  }
}

JWT pitfalls to avoid

  • Never use alg: none — always explicitly set algorithms in verify()
  • Never store JWTs in localStorage — use httpOnly cookies to prevent XSS theft
  • Use short expiry (15m) for access tokens; issue refresh tokens separately
  • Never put sensitive data (passwords, PII) inside the JWT payload — it's only signed, not encrypted

Step 4: Implement Refresh Tokens ​

Short-lived access tokens are great for security, but you don't want users to log in every 15 minutes. Refresh tokens solve this:

javascript
// src/lib/tokens.js
import crypto from 'crypto'
import { saveRefreshToken, findRefreshToken, deleteRefreshToken } from '../db/tokens.js'

// Generate a cryptographically random refresh token
export async function issueRefreshToken(userId) {
  const token = crypto.randomBytes(64).toString('hex')
  const expiresAt = new Date(Date.now() + 7 * 24 * 60 * 60 * 1000) // 7 days

  // ✅ Store a HASH of the refresh token, not the raw value
  const hash = crypto.createHash('sha256').update(token).digest('hex')
  await saveRefreshToken({ userId, hash, expiresAt })

  return token // Return raw token to the client once
}

export async function rotateRefreshToken(rawToken) {
  const hash = crypto.createHash('sha256').update(rawToken).digest('hex')
  const stored = await findRefreshToken(hash)

  if (!stored || stored.expiresAt < new Date()) {
    throw new Error('Invalid or expired refresh token')
  }

  // ✅ Rotate: delete old token and issue a new one (prevents reuse)
  await deleteRefreshToken(hash)
  return issueRefreshToken(stored.userId)
}

Step 5: Add Multi-Factor Authentication (MFA) ​

MFA adds a second layer: even if a password is stolen, the attacker still can't log in without the second factor. We'll implement TOTP (Time-based One-Time Passwords) — the kind used by Google Authenticator, Authy, and 1Password.

bash
npm install otplib qrcode

Generate and return a setup QR code:

javascript
import { authenticator } from 'otplib'
import QRCode from 'qrcode'

export async function setupMFA(user) {
  const secret = authenticator.generateSecret() // Save this to the user's record
  const otpAuthUrl = authenticator.keyuri(user.email, 'SecureApp', secret)
  const qrDataUrl = await QRCode.toDataURL(otpAuthUrl)

  return { secret, qrDataUrl }
  // Show qrDataUrl to user as an <img> for scanning
  // Save secret to DB (encrypted) — only store after user verifies it works
}

Verify a TOTP code at login:

javascript
export function verifyTOTP(token, secret) {
  return authenticator.verify({ token, secret })
}

Updated login flow with MFA:

javascript
// After verifying password:
if (user.mfaEnabled) {
  const { totpCode } = req.body
  const valid = verifyTOTP(totpCode, user.mfaSecret)
  if (!valid) {
    return res.status(401).json({ error: 'Invalid MFA code.' })
  }
}
// Proceed to issue token...

Recovery codes

Always generate and show one-time recovery codes when a user enables MFA. Store bcrypt-hashed versions in your database. If users lose their authenticator device, these are their only lifeline. Generate 10 random 16-character alphanumeric codes.

✅ Chapter 3 Checklist ​

  • [ ] Passwords hashed with bcrypt at cost factor ≥ 12 (or Argon2id)
  • [ ] Plain-text passwords never logged or stored anywhere
  • [ ] Login endpoint doesn't reveal whether an email exists
  • [ ] Timing-safe comparison used — bcrypt.compare always runs
  • [ ] JWTs signed with a strong secret, short expiry (15m)
  • [ ] algorithms whitelist set in jwt.verify()
  • [ ] Access tokens stored in httpOnly cookies, not localStorage
  • [ ] Refresh token rotation implemented
  • [ ] TOTP-based MFA available for users
