Dependency Security Chapter 7 ​

Series Navigation

← Chapter 6: Secrets & Environment Config · Chapter 8: Testing & Auditing →

A typical Node.js project has hundreds — sometimes thousands — of transitive dependencies. Each one is code written by someone else, installed automatically, and run with the same privileges as your own application. Supply chain attacks, where malicious code is injected into legitimate packages, are one of the fastest-growing attack vectors in software today.

The Threat Landscape ​

Real-world supply chain attacks
  • event-stream (2018) — A maintainer handed off a popular package to a stranger. The new maintainer injected code to steal Bitcoin from a specific wallet app. The package had 2M weekly downloads.
  • ua-parser-js (2021) — A developer's npm account was compromised. Malicious versions were published that installed a cryptominer and credential stealer.
  • node-ipc (2022) — A maintainer intentionally added code that deleted files on Russian and Belarusian computers, as a protest. This broke hundreds of projects including Vue CLI.
  • colors & faker (2022) — A maintainer deliberately broke their own packages in protest of free open-source labor, crashing thousands of dependent projects.

The lesson: dependencies are a trust relationship. Every package you install is code that runs on your servers and your users' machines.

Step 1: Audit Your Dependencies Regularly ​

Built-in: npm audit ​

bash
npm audit               # Show all vulnerabilities
npm audit --audit-level high  # Only show high and critical
npm audit fix           # Auto-fix compatible updates
npm audit fix --force   # Fix with potentially breaking major version bumps

The output tells you:

  • The severity (low / moderate / high / critical)
  • The vulnerable package and version range
  • Which of your direct dependencies pulls it in
  • Whether a fix is available

Make auditing part of your CI pipeline

Add npm audit --audit-level high to your CI pipeline. A failing audit should block deployment of new code. This ensures no new high/critical vulnerabilities slip through unnoticed.

Deeper Analysis: npm-audit-resolver ​

For large projects where npm audit output is overwhelming:

bash
npm install -g npm-audit-resolver
npm-audit-resolver

This interactive tool lets you acknowledge, snooze, or act on each vulnerability individually, maintaining an audit-resolve.json file that tracks your decisions.

Step 2: Keep Dependencies Updated ​

Outdated packages are a primary source of known vulnerabilities. Two strategies:

Manual: Check Outdated Packages ​

bash
npm outdated          # Show outdated packages in a table
npm update            # Update within semver range (minor + patch)
npx npm-check -u      # Interactive update picker

Automated: Dependabot or Renovate ​

Both are free tools that automatically open pull requests when your dependencies have updates or known vulnerabilities.

Dependabot — built into GitHub, zero config to enable:

yaml
# .github/dependabot.yml
version: 2
updates:
  - package-ecosystem: "npm"
    directory: "/"
    schedule:
      interval: "weekly"
    open-pull-requests-limit: 10
    # Group minor/patch updates into one PR to reduce noise
    groups:
      minor-and-patch:
        update-types:
          - "minor"
          - "patch"

Renovate — more configurable, supports monorepos and auto-merging:

json
// renovate.json
{
  "$schema": "https://docs.renovatebot.com/renovate-schema.json",
  "extends": ["config:recommended"],
  "automerge": true,
  "automergeType": "pr",
  "automergeStrategy": "squash",
  "packageRules": [
    {
      "matchUpdateTypes": ["patch"],
      "automerge": true
    }
  ]
}

Step 3: Lock Your Dependency Tree ​

A lockfile (package-lock.json or yarn.lock) records the exact version and integrity hash of every package installed. This prevents "works on my machine" problems — and supply chain attacks that inject malicious code into a new minor/patch release.

bash
# Always commit your lockfile
git add package-lock.json
git commit -m "chore: update lockfile"

In CI, use npm ci instead of npm install:

bash
# npm ci:
# - Installs EXACTLY what's in the lockfile (no range resolution)
# - Fails if lockfile is out of sync with package.json
# - Faster than npm install in CI
npm ci

Never use npm install --legacy-peer-deps in CI

This flag bypasses peer dependency resolution and can result in installing different packages than you expect. Fix peer dependency issues properly.

Step 4: Evaluate Packages Before Installing ​

Before adding any new dependency, run a quick evaluation:

Using npx cost-of-modules ​

bash
npx cost-of-modules --no-install --include-dev

Shows the size of each dependency — useful for spotting bloated packages.

Checklist Before npm install ​

bash
# Check download stats and last publish date
npm info <package>

# Check how many dependencies it brings in
npx howmanypeople <package>

Ask these questions:

  • Is it maintained? When was the last commit / publish? Is the repo archived?
  • Is it necessary? Can you implement this functionality yourself in < 20 lines?
  • How many downloads does it have? Widely used packages get more scrutiny.
  • How many dependencies does it pull in? More dependencies = larger attack surface.
  • Does it have a security policy? Check for a SECURITY.md in the repo.

The "is it necessary?" test

left-pad (a package that left-pads a string) once broke thousands of Node projects when it was unpublished from npm. It was 11 lines of code. Ask yourself whether a new dependency is truly worth the risk and maintenance cost before adding it.

Step 5: Use Package Integrity Checking ​

npm verifies packages against their published checksum automatically. You can add an extra layer by pinning exact versions for critical dependencies:

json
// package.json — use exact versions for security-critical packages
{
  "dependencies": {
    "express": "4.21.1",
    "bcrypt": "5.1.1",
    "jsonwebtoken": "9.0.2"
  }
}

Instead of "^4.21.1" (allows any 4.x.x), "4.21.1" installs exactly that version. This prevents a compromised patch release from automatically landing in your app.

Step 6: Protect Against Typosquatting ​

Typosquatting is when an attacker publishes a malicious package with a name similar to a popular one — hoping developers make a typo:

LegitimateTyposquat
expressexpres, expresss
lodashlodahs, 1odash
reactrecat, rect

Defenses:

  • Always double-check package names before installing
  • Use npm install from package.json, not from memory
  • Enable 2FA on your npm account to prevent account takeover
  • In organizations, use a private registry or allowlist (see below)

Use a Private Registry or Allowlist ​

For production applications, consider routing all package installs through a registry proxy like Verdaccio or npm Enterprise that enforces an approved package list:

bash
# .npmrc — proxy all installs through your private registry
registry=https://registry.your-company.com

Step 7: Monitor for New CVEs Continuously ​

Vulnerabilities are discovered in packages after you install them. You need ongoing monitoring, not just a one-time audit:

ToolHow to use
Snyknpx snyk monitor — continuous monitoring with alerts
Socket.devGitHub app that analyzes PRs for supply chain risk
GitHub Dependabot AlertsAutomatic, built into GitHub repositories
OSV Scannerosv-scanner --lockfile package-lock.json
bash
# One-time scan with OSV Scanner (Google's open-source vulnerability DB)
npx @google/osv-scanner --lockfile=./package-lock.json

✅ Chapter 7 Checklist ​

  • [ ] npm audit run and all high/critical vulnerabilities addressed
  • [ ] npm audit --audit-level high added to CI pipeline
  • [ ] Lockfile (package-lock.json) committed to version control
  • [ ] CI uses npm ci instead of npm install
  • [ ] Dependabot or Renovate configured for automated updates
  • [ ] New packages evaluated for maintenance status and necessity before adding
  • [ ] Security-critical packages pinned to exact versions
  • [ ] npm 2FA enabled on all team members' accounts
  • [ ] Continuous vulnerability monitoring enabled (Snyk, Socket, Dependabot alerts)
