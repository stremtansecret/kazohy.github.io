HTTPS & Transport Security Chapter 2 ​

Series Navigation

← Chapter 1: Getting Started · Chapter 3: Authentication & Passwords →

Data traveling between your user's browser and your server crosses dozens of routers, switches, and ISPs. Without encryption, anyone along that path can read, modify, or inject content into the connection. HTTPS is the non-negotiable baseline for every modern web application.

In this chapter you'll learn how TLS works, how to get a certificate for free, how to configure your server correctly, and how to lock browsers into HTTPS forever using HSTS.

What Is TLS? ​

Transport Layer Security (TLS) is the cryptographic protocol that powers HTTPS. It provides three things:

PropertyWhat it means
ConfidentialityTraffic is encrypted — eavesdroppers see only noise
IntegrityAny tampering in transit is detected and the connection is dropped
AuthenticationThe certificate proves you're talking to the real server, not an impersonator

TLS vs SSL

You'll often hear "SSL" used colloquially, but SSL is obsolete and broken. Modern servers use TLS 1.2 at minimum, and you should target TLS 1.3 wherever possible. Never enable SSLv3, TLS 1.0, or TLS 1.1.

Step 1: Get a Free Certificate with Let's Encrypt ​

Let's Encrypt is a free, automated, and open Certificate Authority trusted by all major browsers. The recommended client is Certbot.

On Ubuntu/Debian with Nginx:

bash
sudo apt update
sudo apt install certbot python3-certbot-nginx
sudo certbot --nginx -d yourdomain.com -d www.yourdomain.com

On Ubuntu/Debian with Apache:

bash
sudo apt install certbot python3-certbot-apache
sudo certbot --apache -d yourdomain.com -d www.yourdomain.com

Certbot will automatically edit your web server config to enable HTTPS and set up auto-renewal. Verify it works:

bash
sudo systemctl status certbot.timer   # Should show active
sudo certbot renew --dry-run          # Simulate a renewal

Certificates expire every 90 days

Let's Encrypt certificates are valid for 90 days. Certbot installs a systemd timer or cron job to renew them automatically. Always verify this is working. A lapsed certificate will immediately break your site with a scary browser error.

Step 2: Configure TLS Correctly in Nginx ​

A default Nginx HTTPS config is not always secure. Here's a hardened configuration:

nginx
server {
    listen 443 ssl http2;
    listen [::]:443 ssl http2;
    server_name yourdomain.com www.yourdomain.com;

    # --- Certificate paths (set by Certbot) ---
    ssl_certificate     /etc/letsencrypt/live/yourdomain.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/yourdomain.com/privkey.pem;

    # --- Protocol: only TLS 1.2 and 1.3 ---
    ssl_protocols TLSv1.2 TLSv1.3;

    # --- Strong cipher suites only ---
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:ECDHE-ECDSA-CHACHA20-POLY1305:ECDHE-RSA-CHACHA20-POLY1305;
    ssl_prefer_server_ciphers off; # Let the client pick with TLS 1.3

    # --- Session resumption (performance + security) ---
    ssl_session_timeout 1d;
    ssl_session_cache shared:MozSSL:10m;
    ssl_session_tickets off; # Tickets have forward secrecy concerns

    # --- OCSP Stapling (faster cert validation) ---
    ssl_stapling on;
    ssl_stapling_verify on;
    resolver 1.1.1.1 8.8.8.8 valid=300s;
    resolver_timeout 5s;

    # --- HSTS (covered in Step 4) ---
    add_header Strict-Transport-Security "max-age=63072000; includeSubDomains; preload" always;

    location / {
        proxy_pass http://localhost:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}

Step 3: Redirect All HTTP Traffic to HTTPS ​

Never serve content over plain HTTP. Redirect every request:

nginx
# Redirect HTTP → HTTPS
server {
    listen 80;
    listen [::]:80;
    server_name yourdomain.com www.yourdomain.com;

    # ACME challenge for Certbot renewals
    location /.well-known/acme-challenge/ {
        root /var/www/certbot;
    }

    # Everything else → HTTPS
    location / {
        return 301 https://$host$request_uri;
    }
}

Use 301, not 302

A 301 Permanent redirect tells browsers (and search engines) this is a permanent change. A 302 Temporary redirect will not be cached by browsers, meaning every HTTP visit goes through a redirect round-trip forever.

Step 4: Implement HSTS ​

HTTP Strict Transport Security (HSTS) is an HTTP response header that tells browsers: "From now on, only ever connect to this domain over HTTPS — even if the user types http://."

The header looks like this:

http
Strict-Transport-Security: max-age=63072000; includeSubDomains; preload
DirectiveMeaning
max-age=63072000Remember this for 2 years (in seconds)
includeSubDomainsApply to all subdomains too
preloadRequest inclusion in browsers' hardcoded HSTS list

In your Express app (in addition to the Nginx header above):

javascript
import helmet from 'helmet'

app.use(helmet({
  hsts: {
    maxAge: 63072000,        // 2 years
    includeSubDomains: true,
    preload: true
  }
}))

HSTS Preloading ​

Once HSTS is deployed and working for a while, you can submit your domain to the HSTS Preload List — a list of domains hardcoded directly into Chrome, Firefox, Safari, and Edge. Users who have never visited your site before will still get HTTPS enforced.

HSTS Preload is a one-way door

Once you're on the preload list, removing yourself takes months and requires submitting a removal request that browsers must then ship in updates. Only add the preload directive if you're 100% committed to HTTPS-only for your entire domain and all subdomains — forever.

Submit your domain at hstspreload.org.

✅ Chapter 2 Checklist ​

  • [ ] Auto-renewal verified with certbot renew --dry-run
  • [ ] TLS 1.0 and TLS 1.1 disabled
  • [ ] TLS 1.3 enabled
  • [ ] Weak cipher suites removed
  • [ ] HTTP → HTTPS redirect with 301 in place
  • [ ] Strict-Transport-Security header present on all HTTPS responses
